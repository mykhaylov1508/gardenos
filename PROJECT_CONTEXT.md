## 1. Що змінилось і чому

### Помилка першої версії

Попередні документи відповідали на питання:
> «Як людина, яка **вже знає** як вести господарство, може ефективніше відстежувати дані?»

Реальна потреба:
> «Як людина, яка **не знає** як вести господарство, може навчитись і робити це добре, не тримаючи нічого в голові?»

Це принципово різні продукти. Батьки і дядько тримали все в голові — бо вони це знали і набули досвіду роками. Без вбудованих знань будь-який трекер є просто нотатником, а нотатник для початківця — марний.

### Де це видно у попередніх специфікаціях

- **GardenOS_Специфікація.md** — добре описана CRM для рослин з модулями Ресурсів, Переробки, PDF-звітів, ROI-аналізу. Написана для фахівця або для стартапу з командою. Модуль «Бібліотека знань» взагалі відсутній.
- **GardenOS_Модуль_СезоннийЦикл.md** — набагато ближча до реальних потреб: шкала дозрівання, погодні алерти, нотатки для наступного сезону, прогноз урожаю vs потреби сім'ї. Але реалізована як 7-кроковий конвеєр для агронома, а не для людини що ставить перші кілки під томати.

### Ключовий зсув

| Старий підхід | Новий підхід |
|---|---|
| CRM для рослин | Особистий агроном у телефоні |
| Трекер подій | Система що знає що треба зробити |
| Форми для заповнення | Готові списки дій з поясненнями |
| Аналітика і звіти | «Що зробити сьогодні і чому» |
| Розрахований на знання | Несе знання в собі |

## 2. Філософія: три рівні системи

Вся архітектура продукту будується навколо трьох рівнів:

```
┌─────────────────────────────────────────────────────┐
│  РІВЕНЬ ЗНАНЬ                                       │
│  Що система знає сама                               │
│  Бібліотека: рослини, їх потреби, хвороби, сезони   │
│  Статичні дані, наповнені один раз, підтримуються   │
├─────────────────────────────────────────────────────┤
│  РІВЕНЬ СТАНУ                                       │
│  Що є у твоєму господарстві                         │
│  Ділянка: що посаджено, де, коли, як                │
│  Дані що вводить користувач                         │
├─────────────────────────────────────────────────────┤
│  РІВЕНЬ ДІЙ                                         │
│  Що потрібно зробити                                │
│  Задачі сьогодні, попередження, рекомендації        │
│  Генерується автоматично: Знання + Стан + Погода    │
└─────────────────────────────────────────────────────┘
```

Вся цінність продукту — третій рівень. Він відповідає на єдине щоденне запитання:

> **«Що мені потрібно зробити з господарством сьогодні?»**

Більшість функцій попередньої специфікації живуть у рівні 2 (трекер стану), але не дають рівень 3 (дії). Саме тому вони не створюють реальної цінності для початківця.

## 3. Ключові принципи продукту

**1. Мінімум тримати в голові**
Система знає що і коли потрібно. Користувач отримує готовий список дій — не порожній трекер.

**2. «Чому» поруч з «Що»**
Кожне завдання містить пояснення. «Полий томати» → «Температура вчора була +32°C, ґрунт висихає вдвічі швидше. При спеці ризик скидання зав'язі.» Ти навчаєшся агрономії просто використовуючи продукт.

**3. Навчання через використання**
Через 1–2 сезони використання ти розумієш основи ведення господарства — не через навчальні матеріали, а через щоденні підказки системи.

**4. Поле і план — різний UX**
Польовий режим (телефон на ділянці): 1–2 дії, великі кнопки, без зайвого.
Плановий режим (вдома, планшет/комп'ютер): повний інтерфейс, аналітика, планування.

**5. Дані накопичуються без зусиль**
Записали полив — дані є. Відмітили збір — дані є. Не потрібно «заходити і заповнювати звіти» раз на тиждень.

**6. Smart defaults — система знає більше за тебе до першого запису**
Додав томат → система вже знає інтервал поливу, схему підживлення, типові хвороби, терміни дозрівання. Ти отримуєш задачі автоматично навіть якщо нічого не налаштовував вручну.

---

## 4. Два режими використання

### 📱 Польовий режим (телефон)
Контекст: руки в землі, сонце в очі, 2–3 хвилини.

**Основні дії:**
- Побачити «Що робити сьогодні» — одним поглядом
- Записати полив / підживлення / збір — 2 тапи
- Сфотографувати проблему та отримати AI-аналіз — 3 тапи
- Позначити задачу виконаною — 1 тап

**Що НЕ потрібно у польовому режимі:**
- Планування сезону
- Перегляд графіків і аналітики
- Налаштування системи

### 🖥 Плановий режим (десктоп / планшет)
Контекст: вечір вдома, спокійно.

**Основні дії:**
- Переглянути карту і стан усіх рослин
- Запланувати сезон / грядки / культури
- Переглянути аналітику і підсумки
- Внести детальні нотатки

## 5. Прийняті архітектурні рішення

Ці питання більше не відкриті.

| Питання | Рішення | Причина |
|---|---|---|
| Backend | Supabase-only (без окремого Fastify) | Solo-проект, швидший старт, міграція потім |
| Масштаб | Standalone-продукт (не модуль Village OS) | Фокус і менша складність. Інтеграція пізніше через API |
| Карта | PNG / фото фону без векторного редактора | MVP-просто. Малювання полігонів — Фаза 3 |
| AI | Claude API (claude-3-5-haiku / sonnet) з першого дня | Аналіз фото хвороби — ключова фіча, не можна відкладати |
| AI підхід | Цільовий: фото-аналіз + контекстні підказки | 2 речі зроблені добре > 7 зроблених погано |
| Задачі | Один вигляд: «Сьогодні / Цього тижня / Пізніше» | Канбан і Календар — зайве для початківця |
| Сезонний цикл | Таймлайн-смуга + 3–4 форми (не 7-кроковий конвеєр) | Спрощення без втрати суті |
| Ресурси | Тільки нотатка «купити» без обліку залишків | Повний облік залишків ніхто не веде |
| Переробка | Тільки список ідей («що зробити з томатів»), без калькулятора | Калькулятор — Фаза 3+ |
| Звіти | Тільки auto-сторінка підсумків сезону | PDF/Excel — після 1–2 сезонів реального використання |
| Multi-user | Схема даних закладена (MapMembers), UI — Фаза 3 | Тато може зайти і подивитись |
| Координати карти | Нормалізовані (0.0–1.0 від розміру карти) | Не прив'язані до конкретних пікселів PNG |
| Plant vs PlantingRecord | Обидва з чітким зв'язком (детально у схемі даних) | Plant = іконка на карті; PlantingRecord = сезонна посадка |
| HealthRecords vs HealthObservations | Використовуємо розширену версію (HealthObservations) | Стара таблиця видалена |
| ChangeLog | Видалений як поліморфна таблиця | Замінено на `change_reason` поле в ключових таблицях |
| Zones | Для MVP — текстовий `zone_label` в Plants | Полігони — Фаза 3 |
| Шрифт заголовків | DM Serif Display (закрито) | Органічний, теплий. «Або Nunito» — більше не питання |
| Дизайн-токени | JS-файл tokens.js → імпорт і в Tailwind, і в Konva | Один source of truth для кольорів |
| Мова | Тільки українська для MVP | Фокус на одному ринку |

## 7. Мої пропозиції (нові ідеї поза попередніми документами)

Цей розділ — те, чого не було в жодному з попередніх документів, але що, на мою думку, є критично важливим.

---

### 7.1 Garden Health Score — «Пульс садиби»

**Проблема без нього:** Заходиш в додаток і не розумієш одразу — все окей чи щось горить.

**Рішення:** Один агрегований показник 0–100 що завжди видно на головному екрані.

```
┌──────────────────────────────────┐
│   🌿 Пульс садиби                │
│                                  │
│        78 / 100                  │
│       ████████░░   Добре         │
│                                  │
│  🟢 Полив: в нормі               │
│  🟡 Є 1 незакрита проблема       │
│  🟢 Задач прострочено: 0         │
└──────────────────────────────────┘
```

**Як рахується (автоматично):**
- Частка рослин що не поливались в строк: -10 балів за кожну
- Активні HealthObservations зі статусом `worsening`: -20 балів
- Прострочені задачі: -5 балів кожна
- Не використовувався додаток 5+ днів: -10 балів

**Чому це важливо:** Це емоційний гачок. Число 78 → ти хочеш довести до 90. Число 45 → ти розумієш що щось не так, навіть не читаючи деталей. Це те що змушує відкривати додаток щодня.

---

**Проблема зі звичайним списком задач:** Список задач — нейтральний, нудний, не мотивує.

**Рішення:** Головний екран «Сьогодні» — це не список, а персоналізований ранковий брифінг.

```

│  🌿 GardenOS        📍 Стара Ушиця   ⛅ 24°C         │
├──────────────────────────────────────────────────────┤

│  Доброго ранку! ☀️                                   │
│  Сьогодні буде спекотно — +31°C.                     │
│  Є одна термінова справа.                            │
│                                                      │
│  ┌──────────────────────────────────────────────┐   │
│  │ 💧 Полити томати               ⚠ ТЕРМІНОВО  │   │
│  │ 4 дні без поливу + спека сьогодні = ризик   │   │
│  │ скидання зав'язі. Полий ввечері, 2л/кущ.    │   │
│  │                              [Виконано ✓]   │   │
│  └──────────────────────────────────────────────┘   │
│                                                      │
│  ┌──────────────────────────────────────────────┐   │
│  │ 🔍 Оглянути огірки                          │   │
│  │ Тиждень без огляду. Пік сезону хвороб.      │   │
│  │                       [Переглянути] [✓]     │   │

│  │ 🍅 Черрі Рома — незабаром збирати!          │   │

│  │ протягом тижня. Стеж за кольором.            │   │
│  │                          [Записати збір]     │   │

│                                                      │

**Ключові особливості:**
- Максимум 3–4 картки «сьогодні» — не 15 пунктів
- Одне речення «чому» під кожною задачею — це і є навчання
- Одна кнопка «Виконано ✓» без додаткових форм (деталі за бажанням)
- Вітання з контекстом погоди — відчуття персонального асистента

### 7.3 Мінімальний щоденний звичок — «3 священні дії»

**Проблема:** Додаток з 15 функціями → людина не знає з чого починати → не використовує.

**Рішення:** Виділити 3 дії які дають системі 80% цінності і зробити їх доступними за 2 тапи:

```
1. ЗАПИСАТИ ПОЛИВ
2. ЗАПИСАТИ ПРОБЛЕМУ (з фото)
3. ЗАПИСАТИ ЗБІР
```

Ці три дії завжди доступні з головного екрану через FAB (floating action button). Решта — всередині.

**Чому це важливо для архітектури:** Ці дії — основа `CareEvents` і `HarvestRecords`. Без них система «мовчить». Тому зниження бар'єру для цих трьох конкретних дій — це не UX-прикраса, це функціональна необхідність.

---

### 7.4 Прогресивне розкриття (Beginner Mode)

**Проблема:** Нова людина бачить всі функції одразу → перевантаження.

**Рішення:** Два рівні інтерфейсу.

**Beginner Mode (за замовчуванням для нових юзерів):**
- Спрощені форми (тільки обов'язкові поля)
- Детальні пояснення-підказки скрізь
- Приховані advanced-опції
- Онбординг-підказки перші 2 тижні

**Advanced Mode (вмикається вручну або автоматично через 3 тижні):**
- Всі поля форм
- Розширена аналітика
- Налаштування інтервалів вручну

Технічно: булевий прапорець `user.is_beginner_mode` в профілі. Компоненти форм рендеряться умовно.

---

### 7.5 «Нотатки для наступного сезону» — ключова емоційна фіча

Ця ідея є у Сезонному Циклі але не отримала достатнього акценту. На мою думку — це одна з найцінніших функцій.

**Як це працює:**

1. В будь-який момент сезону ти можеш додати «нотатку для себе-майбутнього»:
   > «Де Барао цього року посадив занадто густо — наступного залишати 70 см між кущами»
   > «Смородина потребує підрізки ще до квітня, бо влітку запізно»

2. При закритті сезону: окремий крок «Підготуй пораду для себе наступного року»

3. **На початку наступного сезону**, при відкритті планування, система показує:

```
┌────────────────────────────────────────────────────┐
│  📖 Твої нотатки з минулого сезону                 │
│                                                    │
│  🍅 Томати: «Де Барао занадто густо — 70 см»       │
│  🫐 Смородина: «Підрізати ДО квітня»               │
│  🥒 Огірки: «Сорт Конкурент — найкраще урожай»     │
│                                                    │
│  [Прийняти до уваги] [Переглянути все]             │
└────────────────────────────────────────────────────┘
```

**Чому це важливо:** Це перетворює ведення записів з «обов'язку» на «лист собі». Це та сама «записна книжка» яку вели батьки — але розумна, яка нагадує в потрібний момент.

---

### 7.6 Onboarding: швидкий перший виграш

**Проблема:** Нова людина відкрила додаток. Якщо вона не отримає цінність за перші 5 хвилин — закриє і більше не повернеться.

**Рішення:** Онбординг в 4 кроки що дають негайну цінність:

```
Крок 1: «Який у тебе регіон?» → вибір з мапи
→ одразу підключається погода

Крок 2: «Додай першу рослину» → вибір з бібліотеки
→ одразу показує: «Наступний полив: через 2 дні»

Крок 3: «Завантаж фото ділянки» → PNG фон карти
→ одразу можна розмістити іконку рослини

Крок 4: «Ось твій перший план на тиждень:»
→ система генерує перші задачі автоматично
```

Користувач виходить з онбордингу з готовим планом на тиждень. Не з порожньою системою.

---

### 7.7 Milestone-моменти

Маленькі святкування щоб зробити процес емоційно залучаючим:

- Перші квіти на томаті → «🌸 Зав'язь з'явилась! Стеж за поливом — це критичний момент»
- Перший збір врожаю → «🎉 Перший урожай сезону! Черрі Рома, 340г»
- Закриття сезону → «🏆 Сезон 2025 завершено. Загальний урожай: 47кг»

Це не «гейміфікація» заради гейміфікації. Це нагадування що ведення господарства — справа з результатами і прогресом, а не лише зобов'язаннями.

## 9. Бібліотека знань (PlantLibrary)

Це те чого не було у жодному попередньому документі, але без чого система є лише нотатником.

### Структура запису

```
PlantLibrary {
  id
  name_uk                    // «Томат»
  name_science               // «Solanum lycopersicum»
  category                   // vegetable / fruit / herb / tree / berry
  icon_type                  // яка іконка на карті

  // Потреби у воді
  watering_interval_days     // 2 (між поливами, норма)
  watering_heat_adjustment   // 0.5 (коефіцієнт скорочення інтервалу при спеці)
  water_per_plant_liters     // 2 (норма за один раз)

  // Підживлення
  fertilizing_schedule_json  // [{week_after_planting: 2, type: "N"},
                             //  {week_after_planting: 5, type: "PK"}]

  // Терміни
  vegetation_days_min        // 85
  vegetation_days_max        // 115
  planting_month_min         // 4 (квітень)
  planting_month_max         // 5 (травень)
  harvest_month_min          // 7
  harvest_month_max          // 9

  // Знання для початківця
  typical_diseases_json      // ["фітофтора", "фузаріоз"]
  companion_plants_json      // ["базилік", "морква"]
  incompatible_plants_json   // ["фенхель", "картопля"]
  rotation_family            // "solanaceae" (не сіяти після своїх)
  beginner_tips_json         // ["Підв'яжи при 30 см", "Пасинкуй щотижня"]

  // Урожайність (для прогнозу)
  yield_per_plant_kg_avg     // 4.0
  yield_per_m2_kg_avg        // 6.0

  // Зберігання (для Фази 3 — переробка)
  storage_methods_json       // ["консервування", "заморозка", "сушіння"]
}
```

### Мінімальний старт: 25 культур

| Овочі | Зелень | Ягоди | Дерева |
|---|---|---|---|
| Томат | Петрушка | Полуниця | Яблуня |
| Огірок | Кріп | Смородина | Груша |
| Перець | Базилік | Малина | Слива |
| Кабачок | Салат | Аґрус | Вишня |
| Картопля | Цибуля зелена | Виноград | |
| Морква | Шпинат | | |
| Цибуля | | | |
| Часник | | | |
| Капуста | | | |
| Буряк | | | |

**Час наповнення:** 2–4 години роботи руками один раз. Потім — поповнення по запиту.

### Як бібліотека «розумніє» через дані користувача

- Якщо в твоєму регіоні томати системно дозрівають на 10 днів пізніше ніж норматив → система адаптує прогнози для твоїх умов.
- Якщо ти записав фактичний урожай «4.2 кг/кущ» — при плануванні наступного сезону система покаже твій власний норматив, а не загальний.

## 10. Уніфікована схема даних

Вирішує всі конфлікти між попередніми документами.

### Таблиці

```sql
-- Користувач
Users
  id, email, name
  garden_name, garden_region
  family_size            -- для калькулятора потреб сім'ї (Фаза 3)
  is_beginner_mode       -- прогресивне розкриття
  preferences_json
  created_at

-- Карта садиби (одна на MVP, потім кілька)
Maps
  id, user_id
  name                   -- «Стара Ушиця»
  bg_image_url           -- PNG план
  width, height          -- розміри зображення для нормалізації координат
  location_lat, location_lng  -- для погодного API
  created_at, updated_at

-- Майбутній multi-user (схема є, UI в Фазі 3)
MapMembers
  id, map_id, user_id
  role                   -- owner / editor / viewer

-- Рослина на карті (постійний об'єкт)
Plants
  id, map_id
  library_id             -- FK → PlantLibrary (NULL якщо кастомна)
  name                   -- назва від користувача
  variety                -- сорт
  category
  icon_type
  pos_x, pos_y           -- 0.0–1.0, нормалізовані
  zone_label             -- текст: «Город», «Сад» (без полігонів для MVP)
  planted_date           -- дата першої посадки (для багаторічних)
  is_perennial           -- TRUE для дерев і кущів
  notes
  created_at, updated_at

-- Сезонна посадка однорічної культури
-- Дерева і кущі (is_perennial=TRUE) → PlantingRecord НЕ мають
PlantingRecords
  id, plant_id
  season_year            -- 2025
  planted_date
  variety                -- сорт цього сезону (може змінюватись)
  quantity_plants
  method                 -- seedling / seed / cutting
  expected_harvest_date  -- planted_date + vegetation_days (авто)
  expected_yield_kg      -- з бібліотеки або вручну
  actual_yield_kg        -- оновлюється автоматично із зборів
  prev_season_crop       -- що росло тут торік (сівозміна, текст)
  notes
  change_reason          -- замінює ChangeLog (чому змінили план)
  created_at, updated_at

-- Уніфікований лог подій догляду
CareEvents
  id, plant_id
  planting_record_id     -- NULL для багаторічних
  event_type             -- watering / fertilizing / pruning /
                         -- treatment / mulching / other
  event_date
  quantity
  unit                   -- l / g / ml
  product_name           -- назва добрива або препарату
  notes
  weather_temp_c         -- авто з API
  weather_condition      -- авто з API
  created_at

-- Спостереження за здоров'ям + AI
HealthObservations
  id, plant_id
  planting_record_id     -- NULL для багаторічних
  observed_at
  description
  symptom_type           -- disease / pest / nutrient_deficiency /
                         -- mechanical / other
  severity               -- 1 мало / 2 помітно / 3 серйозно
  photo_url
  ai_analysis            -- відповідь від Claude Vision
  ai_recommendations     -- що робити
  action_taken           -- що зробили
  status                 -- monitoring / resolved / worsening
  created_at

-- Фотохроніка
Photos
  id, plant_id
  taken_at
  photo_url
  context                -- care_event / health / harvest / general
  context_id             -- id відповідного запису
  created_at

-- Збір врожаю
HarvestRecords
  id, plant_id
  planting_record_id
  harvest_date
  quantity
  unit                   -- kg / g / pcs / l
  quality_first_grade_pct  -- % першого сорту (необов'язково)
  destination            -- consume / process / store / gift / sell
  storage_location       -- «підвал», «холодильник» (текст)
  notes
  created_at

-- Задачі
Tasks
  id
  map_id
  plant_id               -- NULL якщо загальна задача
  title
  description
  reason                 -- ЧОМУ виконати (ключове поле для навчання)
  task_source            -- auto_system / auto_weather / user_created
  due_date
  horizon                -- today / this_week / later
  status                 -- pending / done / skipped
  is_recurring
  recurrence_interval_days
  completed_at
  created_at

-- Погодні алерти
WeatherAlerts
  id, map_id
  alert_type             -- heat / frost / rain / wind / drought
  triggered_at
  weather_data_json
  recommendations_json
  auto_task_id           -- якщо автоматично створив задачу
  user_acknowledged

-- Сезони (для архіву і підсумків)
Seasons
  id, map_id
  year
  start_date, end_date
  status                 -- active / archived
  notes_for_next         -- нотатки для наступного сезону
  summary_json           -- авто-генерується при закритті
```

### Вирішені конфлікти між попередніми документами

| Конфлікт | Рішення |
|---|---|
| Plant vs PlantingRecord | Обидва. Plant = іконка на карті (постійна). PlantingRecord = сезонна посадка однорічної. Дерева/кущі = Plant без PlantingRecord |
| HealthRecords vs HealthObservations | Беремо розширену версію з Сезонного Циклу. Стара таблиця видалена |
| ChangeLog поліморфна таблиця | Видалена. Замінено на `change_reason` в PlantingRecords та `created_at/updated_at` скрізь |
| CareEvents для сезону | Один CareEvents для всього. `season_id` НЕ додаємо — `planting_record_id` дає сезонний контекст |
| ProcessingBatches | Processing переноситься в Фазу 3. Many-to-many через ProcessingInputs |
| Tariffs як таблиця | Для MVP: у `Users.preferences_json`. Окрема таблиця — Фаза 3 |
| Zones як таблиця | Для MVP: `zone_label` — текстове поле в Plants. Полігони — Фаза 3 |

import os
import json
import re
import time
import requests

# --- НАЛАШТУВАННЯ ---
OLLAMA_URL = "http://localhost:11434/api/generate"
# Змініть на свою модель, якщо використовуєте іншу (напр., 'llama3', 'mistral')
MODEL_NAME = "gemma3:12b" 
DATA_DIR = "data"
CULTURES_DIR = os.path.join(DATA_DIR, "cultures")
PROMPT_FILE = os.path.join(DATA_DIR, "prompt_template.txt")
LIST_FILE = os.path.join(DATA_DIR, "cultures_list.txt")

def ensure_dirs():
    """Створює необхідні папки, якщо їх немає"""
    os.makedirs(CULTURES_DIR, exist_ok=True)

def load_prompt_template():
    """Завантажує шаблон промпту"""
    with open(PROMPT_FILE, 'r', encoding='utf-8') as f:
        return f.read()

def load_cultures_list():
    """Завантажує список культур, ігноруючи порожні рядки"""
    with open(LIST_FILE, 'r', encoding='utf-8') as f:
        return [line.strip() for line in f if line.strip()]

def clean_json_response(response_text):
    """
    Очищає відповідь LLM від markdown-обгорток та зайвого тексту,
    залишаючи тільки валідний JSON об'єкт.
    """
    # Видаляємо ```json і ``` з початку і кінця
    text = re.sub(r'^```json\s*', '', response_text, flags=re.IGNORECASE | re.MULTILINE)
    text = re.sub(r'\s*```$', '', text, flags=re.IGNORECASE | re.MULTILINE)
    
    # Знаходимо першу '{' і останню '}' (найнадійніший спосіб для LLM)
    match = re.search(r'\{.*\}', text, re.DOTALL)
    if match:
        return match.group(0)
    return text

def generate_culture_data(culture_name, prompt_template):
    """Відправляє запит до Ollama і повертає очищений JSON"""
    prompt = prompt_template.replace("{CULTURE_NAME}", culture_name)
    
    payload = {
        "model": MODEL_NAME,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.5,        # Трохи вище, щоб уникнути зациклення на одних токенах
            "top_p": 0.9,
            "repeat_penalty": 1.15,    # Запобігає повторенню одних і тих самих фраз ("думаю, думаю...")
            "num_predict": 2000        # Жорстке обмеження довжини, щоб модель не зависала назавжди
        }
    }
    
    try:
        print(f"  🔄 Запит до Ollama ({MODEL_NAME})...")
        response = requests.post(OLLAMA_URL, json=payload, timeout=240)
        response.raise_for_status()
        
        result = response.json()
        raw_text = result.get("response", "")
        
        cleaned_json = clean_json_response(raw_text)
        
        # Перевірка валідності JSON
        data = json.loads(cleaned_json)
        return data
        
    except requests.exceptions.RequestException as e:
        print(f"  ❌ Помилка мережі або Ollama: {e}")
        return None
    except json.JSONDecodeError as e:
        print(f"  ❌ Помилка парсингу JSON. Отриманий текст:\n{cleaned_json[:200]}...")
        return None

def main():
    print("🌱 GardenOS: Генерація бібліотеки рослин через Ollama")
    print("=" * 50)
    
    ensure_dirs()
    
    if not os.path.exists(PROMPT_FILE):
        print(f"❌ Файл промпту не знайдено: {PROMPT_FILE}")
        return
        
    if not os.path.exists(LIST_FILE):
        print(f"❌ Файл списку культур не знайдено: {LIST_FILE}")
        return

    prompt_template = load_prompt_template()
    cultures = load_cultures_list()
    
    print(f"📋 Знайдено {len(cultures)} культур для обробки.\n")
    
    success_count = 0
    skip_count = 0
    
    for culture in cultures:
        # Створюємо безпечне ім'я файлу (напр. "Томат" -> "tomat.json")
        safe_name = culture.lower().replace(" ", "_").replace("-", "_")
        filepath = os.path.join(CULTURES_DIR, f"{safe_name}.json")
        
        # Перевірка: якщо файл вже існує, пропускаємо (можливість відновлення)
        if os.path.exists(filepath):
            print(f"⏭️  Пропущено: {culture} (файл вже існує)")
            skip_count += 1
            continue
            
        print(f"🌿 Обробка: {culture}")
        data = generate_culture_data(culture, prompt_template)
        
        if data:
            with open(filepath, 'w', encoding='utf-8') as f:
                json.dump(data, f, ensure_ascii=False, indent=2)
            print(f"  ✅ Збережено: {filepath}")
            success_count += 1
        else:
            print(f"  ⚠️  Не вдалося згенерувати дані для: {culture}")
            
        # Невелика затримка, щоб не перевантажувати GPU і дати LLM "видихнути"
        time.sleep(2)
        
    print("\n" + "=" * 50)
    print(f"🎉 Готово! Успішно згенеровано: {success_count}, Пропущено: {skip_count}")
    print(f"📁 Перевір папку: {os.path.abspath(CULTURES_DIR)}")

if __name__ == "__main__":
    # Перевірка чи запущений Ollama
    try:
        requests.get("http://localhost:11434", timeout=2)
    except requests.exceptions.ConnectionError:
        print("❌ Помилка: Ollama не запущено! Спочатку виконай 'ollama serve' або відкрий Ollama app.")
        exit(1)
        
    main()
```

import os
from pathlib import Path

# Налаштування: що включаємо, а що ігноруємо
INCLUDE_DIRS = ['frontend/src', 'backend', 'data']
IGNORE_PATTERNS = [
    'node_modules', '.git', 'dist', 'build', '.env', '.env.local', 
    '.env.production', 'package-lock.json', '__pycache__', '.DS_Store',
    'vite.config.js.tsbuildinfo'
]
OUTPUT_FILE = 'PROJECT_CONTEXT.md'

def should_ignore(path):
    return any(ignore in str(path) for ignore in IGNORE_PATTERNS)

def generate_context():
    print(f"🔍 Збираю інформацію для нової нейромережі...")
    
    with open(OUTPUT_FILE, 'w', encoding='utf-8') as out_file:
        out_file.write("# 🌿 GardenOS v2.0 - Контекст проекту для AI\n\n")
        out_file.write("Цей документ містить актуальний стан кодової бази та архітектури.\n\n")
        
        # 1. Структура файлів
        out_file.write("## 📂 Структура проекту (тільки важливі файли)\n```text\n")
        for root, dirs, files in os.walk('.'):
            dirs[:] = [d for d in dirs if not should_ignore(d)]
            files = [f for f in files if not should_ignore(f)]
            
            # Фільтруємо тільки наші папки
            if any(root.startswith(f"./{d}") or root == f".{d}" for d in INCLUDE_DIRS) or root == '.':
                level = root.replace('.', '').count(os.sep)
                indent = ' ' * 2 * level
                out_file.write(f"{indent}{os.path.basename(root)}/\n")
                sub_indent = ' ' * 2 * (level + 1)
                for f in files:
                    out_file.write(f"{sub_indent}{f}\n")
        out_file.write("```\n\n")

        # 2. Вміст файлів
        out_file.write("## 📄 Вміст ключових файлів\n\n")
        
        for root, dirs, files in os.walk('.'):
            dirs[:] = [d for d in dirs if not should_ignore(d)]
            files = [f for f in files if not should_ignore(f) and f.endswith(('.js', '.jsx', '.ts', '.tsx', '.json', '.md', '.sql', '.py', '.txt'))]
            
            if any(root.startswith(f"./{d}") or root == f".{d}" for d in INCLUDE_DIRS) or root == '.':
                for file in files:
                    filepath = os.path.join(root, file)
                    try:
                        with open(filepath, 'r', encoding='utf-8') as f:
                            content = f.read()
                            out_file.write(f"### 📄 `{filepath}`\n")
                            out_file.write("```javascript\n" if file.endswith(('.js', '.jsx')) else "```\n")
                            out_file.write(content)
                            out_file.write("\n```\n\n")
                    except Exception as e:
                        print(f"⚠️ Не вдалося прочитати {filepath}: {e}")

    print(f"✅ Готово! Файл '{OUTPUT_FILE}' створено в корені проекту.")
    print(f"📏 Розмір файлу: {os.path.getsize(OUTPUT_FILE) / 1024:.2f} KB")

if __name__ == "__main__":
    generate_context()
